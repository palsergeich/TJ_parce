## Главная находка серии: ленивые метаданные NTFS

Первый прогон: C++ и Go провалили S6 с идентичной картиной — лаг рос линейно до 30 с,
вставки шли одним бёрстом после остановки писателя (виден в `system.query_log`).
**Причина**: когда писатель держит `.log` открытым (штатное поведение rphost), NTFS лениво
обновляет размер файла в метаданных каталога. Определение роста через обход каталога
(`FindFirstFile`/`DirEntry`) видит замороженный размер:
- Go: рост известных файлов детектился по `WalkDir.Info()` → не читал вовсе;
- C++: рост шёл по хэндлу, но **гейт MIN_FILE_SIZE на дискавери** брал размер из каталога →
  файл «вечно 0 байт» и не брался в слежение;
- Rust: бага не имел (размер по открытому хэндлу с самого начала) — укрепился до
  безусловного read-to-EOF.

Самотесты всех участников баг не ловили: дописывали через открыть-записать-закрыть
(закрытие обновляет метаданные). **Правило зафиксировано в контракте**: размер файла —
только через открытый хэндл (`GetFileSizeEx`/`GetFileInformationByHandle`); обход
каталога — только для обнаружения новых имён.

Вторая находка (само-пойманная обоими): чекпоинт обязан персиститься сразу после каждого
подтверждения вставки — периодический (раз в секунду) давал до 25% дублей на kill -9.
