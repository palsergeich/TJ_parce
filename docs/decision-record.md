## Промежуточный вывод

1. **Лидер — C++ core (`core/`)**: первое место в parse-only, разгромное преимущество по памяти (скользящее окно mmap vs цельное чтение файла у остальных), паритет с Rust на полном корпусе. Дополнительный аргумент вне таблицы: `core/` уже является библиотекой с C ABI — она нужна проекту в любом исходе.
2. **Rust — сильный второй**: лучший на полном корпусе (в пределах погрешности с C++), умеренная память на среднем корпусе; проблема цельночитаемых файлов устранима (чанковый reader, ~день работы).
3. **Go — третий**: 2× медленнее на холодном полном корпусе, худшая память (и в medium — 1.2 ГБ против 550-660 у других). Плюсы — простота и скорость разработки (первый участник, задал контракт).

## Рекомендация к фазе 3 (не финальное решение)

Гибрид: **парсинг — всегда tj_core (C++)**; оболочку агента (служба, чекпоинты, ClickHouse-клиент, /metrics) в фазе 3 строить на **Go или Rust поверх FFI** либо целиком на C++ — выбрать после e2e-серии, где решает качество драйвера ClickHouse и стоимость tail-логики, а не скорость парсера. Если побеждает «без FFI, один язык» — текущие цифры указывают на C++ core как самодостаточного агента.
