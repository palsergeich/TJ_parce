# Карта исходников нормализатора (cpp_parse/count_contexts.cpp)

Инженерная карта кода, составленная при глубоком анализе 2026-07-03. Термины на английском — привязаны к строкам исходника.

## Конвейер

4-stage thread pipeline (cpp_parse/count_contexts.cpp), all stages connected by OptimizedQueue<T> (line 552) — a mutex + condition_variable queue over std::vector with a max_size cap that blocks producers (push() waits while queue_.size() >= max_size_, line 567-577) and a pop_batch(batch, N) consumer API that erases from the vector front (O(n) shift, line 598) and notify_all's producers. finish() unblocks everyone; after finish() push silently DROPS items (line 572).

STAGE 1 — File discovery (main thread): find_log_files (line 1168) recursively walks argv[1] with fs::recursive_directory_iterator, keeps regular files with extension ".log" and size >= MIN_FILE_SIZE = 100 bytes, derives date_prefix from filename via parse_date_from_filename (line 54: "YYMMDDHH.log" -> "20YY-MM-DDTHH:"). Files sorted by size DESC (std::execution::par_unseq if >1000 files and <execution> available, line 1188-1199).

STAGE 2 — Readers: num_readers = min(8, min(files.size(), num_workers)), floor 1 (line 1295). Files are statically pre-assigned to readers by greedy min-accumulated-size balancing (lines 1308-1324; works well because files are size-sorted desc). Each reader OS thread calls reader_thread() (line 799) per file sequentially. reader_thread mmaps the WHOLE file read-only via class MappedFile (line 268): Windows = CreateFileW(GENERIC_READ, FILE_SHARE_READ, FILE_FLAG_SEQUENTIAL_SCAN) + CreateFileMappingW(PAGE_READONLY) + MapViewOfFile of the entire file; POSIX = open(O_RDONLY)+fstat+mmap(PROT_READ, MAP_PRIVATE)+madvise(MADV_SEQUENTIAL). Zero-copy: the reader scans for '\n' with find_char_simd (line 205, AVX2 _mm256_cmpeq_epi8 + movemask + _BitScanForward, prefetch +64 bytes, memchr fallback) and splits the mapping into events (see parsing_rules). Each event is an EventView{const char* data; size_t len} (line 449) pointing INTO the mapping. Events are accumulated into an EventBatch (line 455) = {shared_ptr<MappedFile> file, size_t file_index, string date_prefix, string filename, string file_path, vector<EventView> events (reserve 2000)}. When events.size() >= 2000 the batch is pushed (copied; shared_ptr refcount holds the mapping alive) into OptimizedQueue<EventBatch> event_queue with capacity 300 (line 1282) — this is the primary backpressure point: readers block when 300 batches (~600k events max) are in flight. The mapping is unmapped only when the last EventBatch referencing it is destroyed downstream, so memory is bounded by queue depth but a single huge file is always fully mapped.

STAGE 3 — Parsers: num_parsers = num_workers - num_readers (floor 1, line 1298). parser_thread (line 909) pops up to 10 EventBatches per pop_batch call, iterates events, parses each into a JSON object built in a static thread_local std::string json (line 971; shrink_to_fit when capacity > 1 MB, reserve event.len+256), then output_queue.push(json) — one std::string copy per event into OptimizedQueue<std::string> output_queue capacity 5000 (line 1285). Increments atomic<size_t> events_processed per event.

STAGE 4 — Writer (single thread): writer_thread (line 692) creates parent dirs, opens ofstream binary, writes UTF-8 BOM (EF BB BF, line 717), then loops pop_batch(batch, 5000), appending each JSON string + '\n' to a manual 32 MB write_buffer (line 721) that is flushed to disk whenever it reaches 32 MB, with a final flush after the queue finishes. Alternative sink_thread (line 779) just drains and discards (--no-output benchmark mode).

Shutdown ordering in main (lines 1411-1439): join all readers -> event_queue.finish() -> join all parsers -> output_queue.finish() -> join writer. Total threads = readers(<=8) + parsers + 1 writer + main. Progress: readers bump atomic files_processed and print "Прочитано: N/M файлов (X%)" every 50 files or at the last file (line 1375), guarded by global g_cout_mutex (line 629). Final stats: events_processed, files count, elapsed seconds, total MB, MB/s.

## CLI

count_contexts <путь_к_папке> [workers] [output_file] [--no-output|--no-write|--dry-run]
- argv[1] (required): root directory, scanned recursively for *.log; must exist and be a directory (checked lines 1237-1247, exit 1 otherwise).
- argv[2] (optional): TOTAL worker thread budget split into readers+parsers; default std::thread::hardware_concurrency(); parsed with atoi — non-numeric gives 0 which is coerced to 1; NEGATIVE values wrap to a huge size_t (see bugs).
- argv[3] (optional): output NDJSON file path; default fs::current_path()/"result.jsonl" (line 1332); parent directories auto-created (line 699).
- argv[4] (optional, positional — only read when argc >= 5, so argv[3] must be given too): exactly "--no-output", "--no-write" or "--dry-run" switches the writer to sink_thread (parse everything, write nothing).
Output destinations: parsed data goes ONLY to the output file (never stdout). stdout = human progress/status in Russian; on Windows printed via WriteConsoleW (COUT_UTF8 macro, line 663), interleaved with plain std::cout for numbers. stderr = errors. Exit codes: 1 for usage error, missing/non-directory path, or a top-level exception; 0 otherwise, including "no .log files found" and including writer failures (which actually deadlock, see bugs). No version flag, no help flag, no config file.

## Формат вывода (as-is)

NDJSON: one JSON object per event, LF ('\n') line separator, file begins with a UTF-8 BOM (EF BB BF, writer_thread line 717). Fixed field order per record:
1. "timestamp" (string): batch.date_prefix + time_part, appended RAW without escaping (line 979-981; safe because the event-start mask guarantees the 12-char time is digits/':'/'.'). date_prefix = "20YY-MM-DDTHH:" from the filename (parse_date_from_filename line 54); time_part = "MM:SS.ssssss" (chars before the '-' at offset 12). Result e.g. "2025-11-30T21:00:53.520012". No timezone. If the filename is not 8 leading digits, date_prefix is "" and timestamp degrades to "00:53.520012".
2. "duration" (JSON number, always unquoted, line 982-983): raw digits between '-' and the first ',' (mask-guaranteed >= 1 digit). Units are whatever the TJ wrote (microseconds in modern 1C).
3. "event" (string, json_escape'd): TJ event name (CALL, DBMSSQL, EXCP, ...).
4. "level": emitted as a bare number if is_number_token(level) (line 987), else a quoted escaped string. NOTE (verified 2026-07-03): the "0" initializer at line 960 is dead code — both branches (961-967) overwrite level unconditionally; a header ending right after the event-name comma ("...,EXCP,") yields "level":"" (empty string), and a header with NO second comma drops the whole event at line 953.
5. "filename" (string): base name, e.g. "25113021.log".
6. "file_path" (string): exactly grandparent/parent/filename of the source path (line 836-839), e.g. "Mem\\rphost_26236\\25113021.log" — native separators, so backslashes on Windows (JSON-escaped to \\).
7+ Every TJ property Name=value flattened as a TOP-LEVEL key with the original property name preserved (including colons, e.g. "p:processName", "t:clientID", "Context", "Sql", "Usr"). Value typing rules: (a) value quoted in the source with ' or " -> ALWAYS a JSON string; doubled '' collapses to a single ' in output, doubled "" collapses to a single " (emitted as \\\" in JSON); (b) unquoted value -> emitted as a bare JSON number iff is_number_token(val) (line 478: non-empty, <= 32 chars, only digits/'-'/'.', at least one digit, no leading zero unless "0" or "0.x") AND the field name is not in the always-string set {"SearchString", "Guid", "UUID"} (is_always_string_field line 496); otherwise a quoted string; (c) "Name=" with nothing after '=' at event end -> "". json_escape (line 503) escapes '"', '\\', \b \f \n \r \t, and other control bytes < 0x20 as \\u00XX; all bytes >= 0x20 (including multibyte UTF-8) pass through verbatim, no UTF-8 validation. Multiline values (Context) therefore appear with literal \\r\\n / \\n escape sequences inside one JSON string. Edge cases: duplicate keys are possible (repeated property, or a property literally named "event"/"level"/etc. colliding with base fields); trailing text after the last '=' -less segment is silently dropped; the is_number_token laxness can emit invalid bare tokens like 8.3.22.1704 (see bugs). No record-level metadata (no offset, no line number, no host/pid fields beyond what file_path encodes).

## Правила разбора

Event detection (reader): a new event starts at a line matching is_event_start (line 91), equivalent to regex ^\d{2}:\d{2}\.\d{6}-\d+, — positions 0-1 digits, 2 ':', 3-4 digits, 5 '.', 6-11 six digits, 12 '-', then >= 1 digit then ',' (minimum 15 bytes). The reader walks newline to newline (find_char_simd) and closes the current event exactly when the NEXT physical line matches the mask (lines 856-876); the final event is closed at EOF (line 884). Everything between two event starts — including embedded \r\n inside quoted values — is one event blob, so multiline Context values survive intact WITHOUT any quote-state tracking; a line inside a quoted value that itself happens to match the time mask WILL falsely split the event (the older cpp/ version guarded ,Context= matches with a quote-parity check via count_char_simd; cpp_parse's reader has no such guard for splitting). A file whose first bytes match the mask starts in_event=true immediately (line 828); otherwise leading garbage (e.g. a BOM line) is skipped until the first mask match on a subsequent line.

Event parsing (parser_thread, line 909): trailing '\r'/'\n' of the whole event are trimmed (line 931) — internal CR/LF are preserved and JSON-escaped. Header: first ',' ends the time+duration token; first '-' before it splits time_part / duration_part; next comma-delimited token = event name; next = level. Then a property loop (line 1004): find '='; text before it is the property name; then three value forms:
- Single-quoted (line 1036): scan for '\''; a pair '' is an escape -> emit one ' and continue; a lone ' followed by ',' OR end-of-event is the closing quote; a lone ' followed by anything else is treated as literal data (broken-format tolerance) and scanning continues; unterminated -> value runs to event end.
- Double-quoted (line 1084): scan for '"'; "" is an escape -> emit \\\" and continue; the first lone " closes the value UNCONDITIONALLY (no next-char check, unlike the single-quote branch); unterminated -> to event end.
- Unquoted (line 1122): value runs to the next ',' or event end; typed number-vs-string per is_number_token + is_always_string_field.
After each value one ',' is consumed (line 1145). If no '=' remains, the rest of the event is discarded.

Date derivation: parse_date_from_filename (line 54) requires the first 8 filename chars to be digits, maps YYMMDDHH -> "20YY-MM-DDTHH:" with zero range validation (month 13 accepted); non-conforming names yield an empty prefix, not an error. Input BOM is never stripped. Input encoding is assumed UTF-8 and passed through byte-for-byte.

## Ограничения

- Whole-file memory mapping only: MappedFile::open (line 332) maps the entire file in one view; file size is frozen at open time (no re-stat, no incremental map growth), so data appended after open is never seen — unusable as-is for tailing growing logs. A mapped file stays pinned until every EventBatch referencing its shared_ptr is consumed.
- Windows CreateFileW uses FILE_SHARE_READ only (line 337) — opening a .log currently held open for writing by a 1C rphost will fail with a sharing violation; live-log reading needs FILE_SHARE_READ|FILE_SHARE_WRITE|FILE_SHARE_DELETE.
- Input files: only extension '.log', recursive scan, files under MIN_FILE_SIZE = 100 bytes silently skipped (line 1175). Date/hour comes exclusively from the YYMMDDHH.log filename; wrong or renamed filenames silently produce date-less timestamps. Century hardcoded to '20'. file_path keeps exactly two ancestor directory levels (assumes <logdir>/<process_PID>/<YYMMDDHH>.log layout) with OS-native separators.
- No input BOM handling: a UTF-8 BOM at file start prevents the first line from matching the event mask, so the first event of a BOM-prefixed file is silently dropped. Input assumed UTF-8; bytes are passed through with no validation (invalid UTF-8 propagates into the JSON output).
- Output always gets a UTF-8 BOM prepended (writer_thread line 717) — many NDJSON consumers (jq, ClickHouse, Elastic bulk) reject or mis-parse a BOM before the first record.
- All console text is Russian; on Windows it is emitted via WriteConsoleW (COUT_UTF8, line 650), which only works on a real console — redirecting stdout to a file/pipe loses those messages entirely, and numeric fragments printed via std::cout interleave separately.
- No cancellation mechanism: readers loop to EOF, no stop flag; the only lifecycle is run-to-completion of a directory snapshot (one-shot batch semantics).
- Constants LARGE_FILE_THRESHOLD and CHUNK_SIZE (50 MB, lines 81-82) plus CONTEXT_START/ContextInfo/hash_bytes*/starts_with_pattern_simd/count_char_simd are dead code inherited from the older counter — no chunked processing actually exists in cpp_parse.
- DIFFERENCE cpp/ vs cpp_parse/: E:\git\ТехЖурнал\cpp\count_contexts.cpp (1382 lines) is the older tool with the same MappedFile/OptimizedQueue/SIMD scaffolding but a different purpose: it extracts ONLY ,Context='...' values (validating candidates by apostrophe-parity from event start, count_char_simd line 895, to skip matches inside other quoted values), strips \r, unescapes '' -> ', and counts UNIQUE context strings into per-parser unordered_map<string,size_t> merged at the end; output is a human-readable sorted report result_cpp.txt written next to the input dir's parent (fallback CWD), not NDJSON. Its CLI is only <dir> [workers]; thread split is 50/50 readers/parsers (num_readers = num_workers/2) instead of cpp_parse's min(8,...) readers; its event queue is OptimizedQueue<Event>(1000) with per-event granularity (each Event carries its own shared_ptr) instead of EventBatch(300) with 2000-event packs; it also contains an unused FNV-1a-hash dedupe path (add_context_optimized/extract_contexts_from_buffer, lines 428-547). cpp_parse dropped context counting entirely and became a full event->NDJSON normalizer with filename-derived timestamps, property flattening and typed values.

## Найденные баги

- Writer-failure deadlock: if writer_thread cannot create the output directory or open the file it prints an error and returns (lines 704-714) WITHOUT draining output_queue or signalling anything; parsers then block forever in output_queue.push() once 5000 items accumulate, main never reaches output_queue.finish() (line 1436, after parser join), and the process hangs with exit code never delivered.
- is_number_token (line 478) accepts any <=32-char mix of digits/'-'/'.' with at least one digit: values like 8.3.22.1704 (typical 1C version strings), 1.2.3, 1-2, '1-', '.5', '0.' are emitted as bare unquoted tokens, producing INVALID JSON lines.
- Reader splits events without quote-state tracking (lines 856-876): a multiline quoted value containing a physical line that matches ^\d{2}:\d{2}\.\d{6}-\d+, is falsely split into two corrupt events. The older cpp/ version's apostrophe-parity guard was not carried over to event splitting.
- UTF-8 BOM at the start of an input .log silently drops that file's first event (no BOM skip before the is_event_start check at line 828).
- Negative or garbage argv[2]: atoi result is assigned to size_t (line 1249); e.g. '-2' becomes ~1.8e19, num_parsers = num_workers - num_readers stays astronomically large, and the thread-spawn loop (line 1396) dies attempting billions of threads. Only 0 is guarded.
- POSIX build is broken: MappedFile::open non-Windows branch uses ::open(O_RDONLY) and struct stat (lines 388-398) but only <sys/mman.h> and <unistd.h> are included — <fcntl.h> and <sys/stat.h> are missing, so the Linux path will not compile.
- Duplicate JSON keys: a TJ property named event/level/filename/file_path/timestamp/duration, or the same property repeated in one event, produces duplicate keys in the object; behavior downstream is consumer-defined.
- OptimizedQueue::push silently drops items when finished_ is already set (line 572) — latent data-loss hazard for any future use where finish() can race producers (exactly the situation a tail-mode shutdown creates).
- Single-quote closing heuristic (line 1062) requires the char after the quote to be ',' or event-end; a closing quote followed by anything else (e.g. a stray '\r' mid-event, or malformed 'val'extra) is folded into the value and the parser swallows subsequent properties into it.
- pop_batch erases from the front of a std::vector (line 598) — O(n) memmove per pop under lock; with 5000-string batches on the output queue this is measurable contention, a deque/ring would be strictly better.
- hash_bytes_simd (line 142) is dead code in cpp_parse and its 'AVX2' path is fake SIMD (stores the vector to a stack array and hashes byte-by-byte — FNV-1a is inherently sequential).
- Progress and 'Обработано файлов' totals report files.size() even when individual files failed to mmap (reader just logs to stderr and returns, line 807-811); there is no failed-file count or nonzero exit.

## Что нужно для превращения в ядро агента

- (a) STDOUT STREAMING: replace writer_thread (line 692) with a stdout sink: _setmode(_fileno(stdout), _O_BINARY) on Windows to stop LF->CRLF mangling, DELETE the BOM write (line 717), keep the append-to-buffer loop but flush on a time budget (e.g. every batch or 50-100 ms) instead of only at 32 MB so downstream consumers see records promptly; move ALL human/status output to stderr and kill the WriteConsoleW path (COUT_UTF8, lines 650-671) or gate it on _isatty — today redirected stdout silently loses those messages and would corrupt an NDJSON stream if they went to stdout. Fix the writer-failure path to call a fatal callback / finish the queues instead of returning (deadlock bug, lines 704-714).
- (b) TAIL/FOLLOW MODE: the whole-file mmap snapshot (MappedFile::open line 332) must become either (1) periodic re-GetFileSizeEx + re-CreateFileMapping/MapViewOfFile of the grown region, or simpler (2) buffered ReadFile/pread from a remembered offset into a reusable buffer (EventView then points into that buffer; keep the EventBatch shared_ptr pattern, just holding the buffer instead of the mapping). Critical semantic change in reader_thread: the 'last event' flush at EOF (lines 883-895) must NOT run in follow mode — the trailing bytes after the last confirmed event-start are an incomplete event until the next ^\d{2}:\d{2}\.\d{6}-\d+, line arrives or an idle timeout fires; carry them as the next read's prefix. Open handles with FILE_SHARE_READ|FILE_SHARE_WRITE|FILE_SHARE_DELETE (line 334-342; current FILE_SHARE_READ-only will get sharing violations on live rphost logs). Remembered offsets: persist {volume serial + file index from GetFileInformationByHandle (not path — 1C reuses names), committed_offset}; detect truncation/rotation via size < offset or identity change -> restart at 0; discover new YYMMDDHH.log files via ReadDirectoryChangesW instead of the one-shot find_log_files (line 1168). Offsets may only be checkpointed AFTER durable emission: thread the {file_id, event_end_offset} through EventBatch and have the sink ack the high-water mark — note events can complete out of order across parser threads, so commit min-contiguous offset. Add a stop flag checked in the reader scan loop (line 845) — there is currently no cancellation at all.
- (c) LIBRARY/DLL EMBEDDING (Go cgo / Rust FFI / C++ host): all orchestration lives inline in main() (lines 1221-1475) with globals (g_cout_mutex line 629) — extract a NormalizerPipeline class owning both queues + thread pools with a config struct {num_readers (cap currently hardcoded 8 at line 1295), num_parsers, event_queue_cap=300, output_queue_cap=5000, batch_events=2000}. Replace every std::cerr/COUT_UTF8 site (writer 704-775, reader 808-905, main 1227-1464) with an on_error/on_log callback; REMOVE setup_terminate_handler's std::abort (line 1205-1218) — a DLL must never abort the host — and catch all exceptions at the C ABI boundary returning error codes. C ABI sketch: tj_create(config, sink_fn(void* ud, const char* json, size_t len), ud) / tj_add_file(h, utf8_path, date_prefix_or_null) / tj_follow(h, dir) / tj_drain(h) / tj_stop(h) / tj_destroy(h); all strings UTF-8, sink invoked from parser threads (document that) or funnel through the writer thread for single-threaded delivery — reuse output_queue for that. Make date_prefix an injectable parameter (parse_date_from_filename line 54 becomes a default helper) so hosts can supply full timestamps or timezone offsets. The static thread_local json buffer (line 971) is safe in a DLL as long as the library owns its threads. While refactoring, fix is_number_token (line 478) to a strict JSON-number grammar — Go's encoding/json and Rust serde will reject the current invalid bare tokens like 8.3.22.1704.
- (d) PER-FILE PROGRESS/METRICS: reader_thread already computes lines_processed and events_found (lines 832-833) but discards them — return them, and add bytes_scanned. Today the only counters are global atomics events_processed/files_processed with console prints every 50 files (line 1375). For agent telemetry: (1) attach file_id + end_offset to each EventBatch; (2) per-file atomic counters {events_parsed, records_emitted, parse_skips (each 'continue' at lines 935/940/944/953 is a silently dropped event — count them)}; (3) emit a structured file-completion record when the last batch of a file is consumed — the natural hook is shared_ptr<MappedFile> refcount reaching zero (custom deleter) or an explicit EOF-sentinel batch after line 894; (4) expose event_queue.size()/output_queue.size() (OptimizedQueue::size, line 623) as gauges for backpressure monitoring — in follow mode a blocked stdout consumer backpressures push() all the way to the readers, which is the desired behavior but must be observable. Emit metrics as periodic NDJSON control records on stderr (never stdout, to keep the data stream pure) in the form {"type":"progress","file":...,"offset":...,"events":...}.
- PORTING PRIORITIES if rewriting the core in Go/Rust instead of wrapping: the essential IP is small — is_event_start (line 91), the split-on-next-event-start reader loop (lines 845-895), the header+property state machine with the '' / "" doubling rules (lines 926-1147), is_number_token + is_always_string_field typing (478-500), and parse_date_from_filename (54). Everything else (queues, SIMD memchr, mmap wrapper) has superior native equivalents (Go channels + bufio, Rust crossbeam + memchr crate).
