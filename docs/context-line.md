# context_line — значимая строка контекста

Статус: v1 (реализовано в Go-агенте, rich-схема). Код: `agents/go/internal/chsink/ctxline.go`.
Опция конфигурации: `context_skd_smart` (по умолчанию **включена**), CLI: `--context-skd-smart true|false`.

## Две аналитики из одного свойства Context

Контекст 1С — стек вызовов (строки «Модуль : строка : код», разделённые переводами строк).
В rich-схеме `tj.events` он живёт двумя представлениями:

| Колонка | Что это | Зачем |
|---|---|---|
| `context` + `context_hash` | полный стек + cityHash64 сырого текста | группировка одинаковых стеков (agg_context), поиск |
| `context_line` | «значимый отпечаток» — одна строка стека | человекочитаемая метка группы: по ней инженер узнаёт виновника |

`context_hash` — ключ непрерывности с загруженными данными: **ни одно правило
`context_line` не трогает `context` и `context_hash`**. Опция меняет только метку.

## Стандартное правило (как у импортёра)

`context_line` = последняя непустая строка `Context`: все `\r` выбрасываются
(в том числе внутри строки), деление по `\n`, берётся последняя непустая;
пробелы/табы не обрезаются. Эквивалент выражения импортёра
`arrayLast(x -> x != '', splitByChar('\n', replaceAll(ctx, '\r', '')))`.

## Правило СКД (`context_skd_smart`, по умолчанию включено)

Проблема: у операций компоновки данных хвост стека — универсальная обвязка
(`ОбщийМодуль.КомпоновкаДанных.Модуль : 438 : ПроцессорВывода.Вывести(...)`),
одинаковая для всех отчётов конфигурации. По последней строке не видно,
КАКОЙ отчёт виноват; настоящий виновник (модуль `Отчет.*`/`Обработка.*`)
находится выше по стеку.

### Триггер

Правило срабатывает, только если **стандартная значимая строка** (последняя
непустая) содержит вызов одного из методов (имя + необязательные пробелы/табы +
`(`; регистрозависимо, как пишет платформа):

| Метод (рус.) | Метод (англ.) | Доп. условие (страж) |
|---|---|---|
| `Вывести(` | `Output(` | — |
| `ВывестиЭлемент(` | `OutputItem(` | — |
| `НачатьВывод(` | `BeginOutput(` | — |
| `Инициализировать(` | `Initialize(` | строка содержит `Компоновк` или `DataComposition` |

Токенные границы обязательны: символ **перед** именем — не буква/цифра/`_`
(`ПереВывести(` — не триггер), символ **после** имени — пробелы/табы и `(`
(`ВывестиВДокумент(` — не триггер).

Страж у `Инициализировать`/`Initialize`: имя универсально (инициализируют что
угодно), поэтому триггером оно считается только на СКД-строке — подстрока
`Компоновк` покрывает `ПроцессорКомпоновкиДанных.Инициализировать(...)`,
`МакетКомпоновки`, модуль `КомпоновкаДанных`; `DataComposition` — английские
конфигурации. `Объект.Инициализировать(Параметры)` без признака СКД триггером
не является.

### Выбор виновника

При сработавшем триггере стек сканируется **снизу вверх** и берётся первая
встреченная (то есть самая глубокая) строка, чей модуль — первый сегмент до
` : ` — начинается с одного из префиксов:

- `Отчет.` | `ВнешнийОтчет.` | `Обработка.` | `ВнешняяОбработка.`
- `Report.` | `ExternalReport.` | `DataProcessor.` | `ExternalDataProcessor.`

Точка в префиксе гарантирует совпадение именно первого сегмента
(`ОтчетНедели : …` не совпадает). Ведущие пробелы/табы строки допускаются;
возвращается строка целиком в конвенциях стандартного правила (сырая строка,
без `\r`, без обрезки пробелов). Если такой строки нет — **фолбэк** на
стандартное правило (последняя непустая строка).

### Пример (эталон владельца)

```text
ОбщийМодуль.ДлительныеОперации.Модуль : 1390 : ВызватьПроцедуру(...)
…
Отчет.МотивацияАдминистраторовСОтсмотром.МодульОбъекта : 175 : ...
Отчет.МотивацияАдминистраторовСОтсмотром.МодульМенеджера : 53 : ...
Отчет.МотивацияАдминистраторовСОтсмотром.МодульМенеджера : 280 : КомпоновкаДанных.СформироватьОтчет(...)
ОбщийМодуль.КомпоновкаДанных.Модуль : 210 : СформироватьОтчетТаблицаЗначений(...)
ОбщийМодуль.КомпоновкаДанных.Модуль : 438 : ПроцессорВывода.Вывести(ПроцессорКомпоновкиДанных, Истина);
```

Стандартное правило даёт строку `438 : ПроцессорВывода.Вывести(...)` (обвязка);
правило СКД — строку `Отчет.МотивацияАдминистраторовСОтсмотром.МодульМенеджера : 280 : …`
(самая глубокая `Отчет.*`).

## Конфигурируемость и охват

- Ключ `context_skd_smart: true|false` в `tj-agent.yaml`; явный CLI-флаг
  `--context-skd-smart true|false` перекрывает файл. Действует в обоих режимах
  Go-агента с rich-схемой (batch-ingest и follow).
- Списки триггеров и префиксов — данные уровня пакета
  (`skdTriggers`/`skdModulePrefixes` в `ctxline.go`); в v1 из конфига не
  переопределяются.
- NDJSON-путь агента (golden-контракт) не затронут: `context_line` существует
  только в rich-схеме.
- Архивный импортёр `deploy/importer/import-jsonl.ps1` считает `context_line`
  выражением ClickHouse по стандартному правилу — правило СКД он НЕ применяет
  (известное расхождение архивного и онлайн-пути; синхронизация — отдельным
  решением).
