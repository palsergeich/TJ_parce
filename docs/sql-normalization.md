# Нормализация SQL — спецификация правил v1

Единственный источник истины правил нормализации текстов SQL техжурнала.
Реализации обязаны воспроизводиться из этого документа; эталонная —
`agents/go/internal/sqlnorm` (константа `RulesVersion = 1` синхронизирована
с версией правил здесь). Изменение любого правила = новая версия документа
и `RulesVersion`, см. [§9 Версионирование](#9-версионирование).

Статус: **v1, диалект MSSQL (DBMSSQL)** — принята владельцем 2026-07-10.

## 1. Задача

Запросы должны агрегироваться корректно. Сырой `sql_text` бесполезен как
ключ группировки: значения-литералы и приложенные значения параметров делают
почти каждое выполнение уникальным (живой корпус: 14 956 392 событий DBMSSQL —
6 276 312 уникальных `sql_hash`). При этом **число параметров — критичный
сигнал**: IN-список из 500 элементов и из 5 — разные планы, склеивать их
без счётчика нельзя.

Решение (требования владельца, финал):

- значения ослепляются в `?`, **число параметров сохраняется как данные**
  (`param_count`, в роллапе — бакеты);
- извлечённые значения складываются в соседнюю колонку `sql_params`
  **позиционно**: i-й элемент массива = i-й `?` нормы (до схлопывания
  списков, см. R8);
- сырой `sql_text` не изменяется и остаётся в своей колонке (drill-down);
- норм-текст **не хранится**: он детерминированно восстанавливается
  нормализатором из любого сырого образца группы (см. §6).

## 3. Выходы нормализации

Для события rich-схемы с непустым `sql_text` (первый непустой из
`Sql | Query | Sdbl` — правило импортёра):

| Колонка | Тип | Семантика |
|---|---|---|
| `sql_norm_hash` | `UInt64` | `cityHash64(норм-текст)` (go-faster/city == ClickHouse бит-в-бит). `0` — нет `sql_text` либо нормализация выключена |
| `param_count` | `UInt16` | `min(len(sql_params), 65535)`; насыщение считается метрикой `tj_agent_sqlnorm_saturated_total`, реальная длина всегда `length(sql_params)` |
| `sql_params` | `Array(String)` | значения параметров в порядке `?` нормы до схлопывания списков |

Пустой `sql_text` → все три поля нулевые (пустой массив). Хэш пустой нормы
(вырожденный текст из одного хвоста) — `cityHash64('') = 11160318154034397263`,
не 0: «текст был, но свёлся к пустоте» отличимо от «текста не было».

## 4. Правила v1 (диалект MSSQL)

Нормализация — один проход по байтам слева направо. UTF-8 безопасен: все
значимые символы — ASCII, байты ≥ 0x80 всегда часть идентификатора.
Первое подошедшее правило поглощает токен целиком.

### 4.1 Литералы → `?` (захватываются в params)

- **R1. Строки.** `'...'` → `?`; экранирование `''` расклеивается: значение
  захватывается без кавычек, `''`→`'`. Юникод-префикс: `N'...'` (и `n'...'`)
  → `?`, `N` поглощается, если он НЕ хвост идентификатора (слева не
  идентификаторный символ). Незакрытая строка (обрыв текста) — до конца
  текста. Пример: `y = 'it''s'` → `y = ?`, params `["it's"]`.
- **R2. Числа.** Токен не внутри идентификатора: `цифры[.цифры][eE[±]цифры]`,
  допускаются `.5` и `5.`; экспонента поглощается только если за `e/E`
  (и опциональным знаком) стоит цифра. Значение захватывается «как написано».
  **Знак** (`-`/`+`) — часть литерала только в однозначно унарном контексте:
  норма пуста либо последний значащий символ нормы — `( , = < > + - * / %`.
  `x = -5` → `x = ?` (`-5`); `x - 5` → `x - ?` (`5`); `SELECT -5` →
  `SELECT -?` (`5`) — минус после идентификатора не однозначен, хэш
  детерминирован в обе стороны. Цифры внутри идентификаторов
  (`T2`, `_Fld25343`, `Поле1`) не трогаются: идентификатор поглощается
  целиком правилом R11.
- **R3. Hex/binary.** `0x`/`0X` + hex-цифры → `?`, захват как написано
  (`0x8145A8…`). Со знаком в унарном контексте — вместе со знаком.

### 4.2 Структурное ослепление (НЕ захватывается)

- **R4. Временные таблицы.** `#` в начале токена: буквенный префикс имени
  сохраняется, **с первой цифры остаток имени отбрасывается**:
  `#tt123` → `#tt`, `##g7x` → `##g`. В DBMSSQL-корпусе встречается только
  `#tt\d+`; для прочих форм (`#T<hex32>` в EXCPCNTX) правило даёт стабильный,
  но грубый префикс (`#Te4ea…` → `#Te`) — деградация принята, §8.4.
- **R5. RPC-параметры.** `@P<цифры>` → `@P` (регистр `P` сохраняется).
  Прочие `@`/`@@`-имена копируются целиком (цифры — часть имени: `@Var7`).
  Значения RPC-параметров, когда они присутствуют (хвост `sp_executesql`),
  являются литералами и ловятся R1–R3.

### 4.3 Плейсхолдеры и хвост значений

- **R6. Хвост `p_N:`.** Строка текста (после `\n` либо с первого байта),
  начинающаяся `p_<цифры>:` + опциональный пробел, открывает терминальный
  блок значений. Значение — до конца строки «как написано» (`0N`, `FALSE`,
  `20251129235959`, `0x…`; завершающий `\r` отбрасывается); значение,
  начинающееся с `'`, разбирается кавычко-осознанно по R1 (может содержать
  переводы строк), остаток строки за закрывающей кавычкой игнорируется.
  Пустые строки между значениями пропускаются. Весь блок **вырезается из
  нормы**. Строки внутри строковых литералов тела хвостом не считаются
  (лексер их не видит). Строка не по формату прекращает блок и возвращает
  управление лексеру (в корпусе не встречается).
- **R7. Плейсхолдеры тела.** Байт `?` вне литералов — плейсхолдер: в норме
  остаётся `?`, в params резервируется слот. По завершении прохода слоты
  заполняются значениями хвоста R6 по порядку: k-е `?` тела ← k-я строка
  хвоста. Недостача значений → слот получает литерал `"?"` (маркер
  «значение неизвестно»); излишек значений хвоста отбрасывается. Оба случая
  в корпусе отсутствуют (§2), но поведение фиксировано.

### 4.4 Схлопывание списков

- **R8. IN/VALUES-списки.** Внутри скобочной группы, открытой сразу после
  ключевого слова `IN` или `VALUES` (регистронезависимо; `IN(` без пробела —
  тоже), последовательность `?, ?` схлопывается в `?` — норма пишет
  `IN (?)` для списка любой длины, **все значения остаются захваченными**,
  `param_count` хранит истинную длину. Соседние конструкторы строк
  `VALUES (?),(?)` схлопываются в `VALUES (?)` (продолжение `,(` после
  закрытой VALUES-группы наследует флаг). Смешанные списки схлопываются
  частично — только соседние пары плейсхолдеров: `IN (1, T1.b, 3)` →
  `IN (?, T1.b, ?)`. Подзапрос `IN (SELECT …)` не затрагивается (внутри
  группы нет соседних `?`). Группы глубже 64 уровней скобок списками не
  считаются. Функциональные скобки (`ISNULL(?, ?)`, `NUMERIC(27, 3)`) НЕ
  схлопываются — правило действует только после `IN`/`VALUES`.
- Следствие для инвариантов: в норме число `?` ≤ `len(sql_params)`;
  равенство — когда схлопывания не было.

### 4.5 Прочее

- **R9. Пробелы.** Серии ` \t\r\n` → один пробел; ведущие/замыкающие
  обрезаются. Хэш устойчив к переформатированию текста вне литералов.
  Внутри строковых литералов и `[скобочных]` идентификаторов пробелы —
  часть значения/имени.
- **R10. Регистр не сворачивается.** 1С генерирует стабильный регистр;
  case-folding не окупается и ломал бы читаемость нормы. Документированное
  решение v1.
- **R11. Идентификаторы.** Начало: латиница, `_`, `$`, байт ≥ 0x80;
  продолжение: те же + цифры + `#` + `@`. Поглощаются целиком (цифры внутри
  не литералы). `[скобочные]` копируются дословно до `]` (`]]` —
  экранирование). `"` — обычный символ (копируется), см. §8.2.
- **R12. Всё остальное** (операторы, `{}`, `;`, `&`, `.`) копируется как
  есть. ODBC-эскейпы дают стабильные формы: `{ts '2024-02-15 00:00:00'}` →
  `{ts ?}`.

## 5. Инварианты (закреплены тестами пакета)

- **P1** идемпотентность текста: `normalize(norm).text == norm.text`
  (и хэш неподвижен);
- **P2** повторная нормализация нормы не извлекает ни одного значения —
  только маркеры `"?"` (в норме не осталось литералов);
- **P3** `count('?', norm) ≤ len(params)`; равенство без схлопывания R8;
- **P4** детерминизм: повторный прогон того же входа — тот же результат;
- **P5** пробельные мутации тела вне литералов не меняют ни норму, ни params;
- **P6** в норме нет апострофов.

## 6. Восстановление имён параметров и drill-down

Норм-текст не хранится; имена и контекст параметров восстанавливаются из
любого сырого образца группы — сырец и params одного события выровнены
позиционно.

Опорный запрос (образец группы):

```sql
SELECT sql_text, sql_params, param_count
FROM tj.events
WHERE sql_norm_hash = {H:UInt64}
LIMIT 1;
```

i-й элемент `sql_params` соответствует i-му литералу/плейсхолдеру сырца в
порядке слева направо (для текстов с хвостом `p_N` — inline-литералы и `?`
вперемешку, в порядке появления в теле). Имя видно в тексте слева от
позиции: `WHERE T1._Fld123 = ?` → params[i] — значение `_Fld123`.

Распределение значений конкретного параметра по группе (пример: 3-й):

```sql
SELECT sql_params[3] AS v, count() AS c
FROM tj.events
WHERE sql_norm_hash = {H:UInt64}
GROUP BY v ORDER BY c DESC LIMIT 20;
```

Поиск событий группы с конкретным значением любого параметра:

```sql
SELECT ts, duration_us, usr, sql_params
FROM tj.events
WHERE sql_norm_hash = {H:UInt64} AND has(sql_params, '0x8145A8494DAAE3A311EE9A7F33D24A23')
ORDER BY duration_us DESC LIMIT 50;
```

Точный норм-текст при необходимости воспроизводится нормализатором из
`sql_text` образца (гарантия P4/§7-верификация: пересчёт 94 312 уникальных
текстов дал 0 расхождений с записанными агентом хэшами и params).

## 7. Хранение и роллап

DDL: свежая установка — `deploy/clickhouse/init/001_schema.sql`; действующий
сервер — идемпотентная миграция `deploy/clickhouse/init/002_sql_norm.sql`
(только аддитивные ALTER/CREATE; применение и порядок — в шапке файла;
применена к живому серверу 2026-07-10, 121,5 млн исторических строк получили
нулевые значения новых колонок — историю при желании можно донормализовать
переимпортом).

- `tj.events` += `sql_norm_hash UInt64`, `param_count UInt16 CODEC(T64, ZSTD(1))`,
  `sql_params Array(String) CODEC(ZSTD(3))`, bloom_filter-индекс
  `idx_sqlnorm_hash`.
- Роллап `tj.agg_query2` (AggregatingMergeTree, ключ
  `(dbms, hour, sql_norm_hash, param_bucket)`, TTL 400 дней) + MV
  `mv_agg_query2` (`WHERE sql_norm_hash != 0`). Бакеты `param_bucket`:
  `'0' | '1' | '2-10' | '11-100' | '101-1000' | '1000+'` — IN-500 и IN-5 не
  склеиваются даже внутри одного хэша нормы. Агрегаты: `cnt/dur_sum/rows_sum`
  — `SimpleAggregateFunction(sum)`, `dur_max` — `(max)`, `dur_q` —
  `quantilesTDigest(0.5, 0.95, 0.99)`, `params_avg` — `avg(UInt16)`.
- **Находка (перенесена в долги фазы):** плоские `UInt64`-счётчики в
  AggregatingMergeTree теряют значения при слияниях — на живом `tj.agg_query`
  `sum(cnt)` = 9,76 млн при 14,96 млн фактических событий (−35 %).
  `agg_query2` использует SimpleAggregateFunction и корректен
  (проверено: `sum(cnt)` == счётчику событий бит-в-бит); старые
  `agg_query`/`mv_agg_query` не тронуты — параллельная работа до миграции
  дашбордов, затем чинить/выводить отдельной задачей.
- Агент: нормализация включена по умолчанию в rich-схеме; `sql_norm: false`
  в YAML-конфиге выключает вычисление И убирает три колонки из INSERT
  (совместимость с tj.events без миграции). Bench-схема и NDJSON-пути не
  затронуты (golden byte-exact).

## 8. Ограничения v1 (документированные деградации)

1. **Комментарии** `--`/`/* */` не обрабатываются: в корпусе вне строковых
   литералов их нет (§2). Появится корпус с комментариями — v2.
2. **Двойные кавычки** — обычный символ. В DBMSSQL `"` встречается только
   внутри `'...'`-строк; в EXCPCNTX (2983 события) `"..."`-строки остаются
   в норме неослеплёнными. Кандидат v2: диалект с `"..."`-строками
   (EXCPCNTX/1C-Query file-mode).
3. **`sp_executesql`**: внутренний запрос — строковый литерал → вся группа
   `{call sp_executesql(?, ?, ?)}` склеивается в один хэш нормы независимо
   от внутреннего текста (значения остаются в params). 529 событий (0,004 %).
   Кандидат v2: рекурсивная нормализация первого аргумента.
4. **`#T<hex32>`** (EXCPCNTX): R4 оставляет от имени префикс до первой цифры —
   имена с ведущей буквой hex дают `#Te`, с цифрой — `#T`; группировка
   грубее идеальной, стабильность в пределах текста сохранена.
5. **QERR/SDBL (1С-язык)** нормализуются MSSQL-правилами: числа и
   `'...'`-строки ослепляются, `"..."`-строки и `&Параметры` остаются как
   есть (структурно стабильны). Полноценный 1C-Query-диалект — v2.
6. **TOP N / OFFSET / FETCH** — числа ослепляются (R2): `TOP 1` и `TOP 1000`
   дают один хэш нормы; различие доступно через `sql_params`/`param_count`.
7. Вне рамок v1 по решению владельца: «причесать» `Context`, разбор
   TLOCK-`Regions` — явно отложены.

## 9. Версионирование

Правила версионируются целиком: любое изменение → v2 этого документа,
`RulesVersion = 2` в `agents/go/internal/sqlnorm`, синхронное обновление
фикстур. Смена версии меняет `sql_norm_hash` части текстов → группы в
`tj.events` рвутся на границе выката. Процедура: зафиксировать дату выката;
при необходимости непрерывности — офлайн-пересчёт колонки по `sql_text`
(норм-текст не хранится, пересчёт всегда возможен). Хэш-функция —
cityHash64 — не меняется никогда (правило проекта №2).

## 10. Производительность

Микробенч пакета (`go test -bench . ./internal/sqlnorm/`,
Ryzen 7 8700F, один поток):

| Вход | нс/оп | пропускная | норм./с на ядро |
|---|---|---|---|
| медиана корпуса 277 Б | 406 | 547 МБ/с | 2,46 млн/с |
| типовой ~0,8 КБ (тело+хвост+IN) | 1471 | 553 МБ/с | 680 тыс./с |
| потолок 3 МБ | 5,3 мс | 594 МБ/с | линейно |

Целевая планка «≥ 1 млн/с на ядро на ~1 КБ» на реальной форме текста не
достигнута (680 тыс./с из-за разбора хвоста и захвата params) — стоимость
задокументирована и в бюджете агента незаметна: на полном корпусе SQL несут
~12 % событий (медиана 277 Б → ~0,4 мкс), при 500 тыс. строк/с это ~61 тыс.
