# Спецификация формата NDJSON нормализатора ТЖ — v1.0 (draft, ревизия 2)

Единственный источник истины о формате обмена «нормализатор → приёмник/БД».
Все три агента bake-off (Go/Rust/C++) обязаны выдавать байт-в-байт этот формат.
Основана на фактическом поведении `cpp_parse/count_contexts.cpp` (см. [normalizer-source-map.md](normalizer-source-map.md)) **плюс зафиксированные ниже исправления** — текущий exe им ещё не полностью соответствует. Ревизия 2: учтены результаты адверсарной сверки спецификации с кодом (2026-07-03).

## 1. Контейнер

- Поток записей: один JSON-объект на строку. LF (`\n`) — **терминатор** каждой записи, включая последнюю (текущий exe так и делает; для golden-сравнения хвостовой LF обязателен).
- Кодировка UTF-8. **Без BOM в начале потока** (решение v1.0; текущий exe пишет BOM — баг KI-7).
- Входной BOM (EF BB BF) в начале `.log` файла **пропускается**; первое событие BOM-файла не теряется (текущий exe теряет — баг KI-6).
- Каждая строка — валидный JSON по RFC 8259. Валидатор (`json.loads` / `serde_json` / `encoding/json`) обязан принимать каждую строку.

## 2. Модель события ТЖ

### 2.1. Разрезание потока на события

Решение о начале нового события принимается **только** по префиксу физической строки:

```
^\d{2}:\d{2}\.\d{6}-\d+,
```

(минимум 15 байт: `ММ:СС.мммммм-Д,`). Всё, что идёт после этой запятой, на сплит **не влияет** — `10:00.000000-5,мусор` тоже начинает событие; отбраковка происходит позже, при разборе заголовка (§6).

Событие заканчивается там, где начинается следующая строка-маска (или EOF). Всё между — одно событие, включая переводы строк внутри значений (многострочный `Context`, `Locks` до ~3.4 МБ).

Перед разбором события **все хвостовые `\r`/`\n` события обрезаются** (внутренние CR/LF сохраняются и экранируются в JSON). Поэтому значение последнего свойства никогда не содержит перевода строки, а пустые строки между событиями исчезают.

**Known issue KI-1:** строка внутри кавычек, сама совпадающая с маской, ложно разрезает событие. Решение: проверка чётности кавычек при сплите (как в старом `cpp/`), фиксируется golden-кейсом `mask_inside_quotes` (до починки — expected-fail).

### 2.2. Заголовок события

После маски событие разбирается как:

```
ММ:СС.мммммм-Длительность,Событие,Уровень[,Имя=Значение]*
```

**Дата и час** берутся из имени файла `YYMMDDHH.log`, **минуты/секунды/микросекунды** — из строки события.

- Нет второй запятой (после имени события) → событие **отбрасывается со счётчиком** `parse_skips` (§6).
- Заголовок `...,EXCP,` (после второй запятой пусто) → `level` = пустая строка `""`.
- Свойств может не быть вовсе — запись из одних базовых полей валидна.

## 3. Поля записи

Фиксированный префикс (в этом порядке):

| # | Поле | Тип | Правила |
|---|------|-----|---------|
| 1 | `timestamp` | string | `20YY-MM-DDTHH:` (из имени файла) + `MM:SS.ssssss` (из события). Без таймзоны — локальное время сервера-источника. Правило извлечения даты: **первые 8 символов имени файла — цифры** (суффикс не проверяется, диапазоны не валидируются: «месяц 13» пройдёт; век зашит `20`). Первые 8 символов не цифры → префикс пустой, `timestamp` деградирует до `MM:SS.ssssss`; агент обязан считать такой файл аномалией и репортить. |
| 2 | `duration` | number | Микросекунды, 64-бит (в корпусе есть значения до 17.5×10⁹). Текущий exe копирует цифры сырьём — ведущие нули (`-007,`) дают невалидный JSON; **решение v1.0: duration печатается канонически, без ведущих нулей** (часть KI-2). |
| 3 | `event` | string | Имя события как в ТЖ (регистр сохраняется: `CALL`, `Context`). |
| 4 | `level` | number \| string | Число, если проходит грамматику числа (§4), иначе строка. Пустой уровень → `""`. |
| 5 | `filename` | string | Базовое имя: `25113021.log`. |
| 6 | `file_path` | string | Ровно два уровня предков: `<коллекция>\<process_pid>\<файл>.log`. Если предков меньше двух — соответствующая часть пустая. Разделители нативные для ОС агента (Windows `\\` в JSON-экранировании). |

Далее — все свойства события `Имя=Значение` как top-level ключи с сохранением оригинального имени (включая двоеточия: `p:processName`, `t:clientID`).

- Имя свойства = всё до следующего `=`, поэтому сегмент без `=` приклеивается к следующему ключу: `Foo,Bar=1` → ключ `"Foo,Bar"`.
- Хвост события, в котором больше нет `=`, **молча отбрасывается** (v1.0: допустимо; желательно считать в `parse_skips` в v1.1).

Планируемые дополнительные поля v1.1 (для агента; приёмник обязан игнорировать незнакомые поля): `collection` (имя коллекции логов), `src_offset` (байтовое смещение события в файле), `host` (имя сервера-источника).

## 4. Типизация и разбор значений

### 4.1. Значения в кавычках

Значение в кавычках (`'...'` или `"..."`) → **всегда** JSON-строка. Правила закрытия (обязательны для байт-точного порта, они несимметричны):

- **Одинарные кавычки:** `''` — экранирование, даёт один `'` в выходе. Одиночная `'` закрывает значение **только** если за ней идёт `,` или конец события; иначе она считается литеральными данными и сканирование продолжается (KI-10: так «глотаются» последующие свойства при кривом формате).
- **Двойные кавычки:** `""` — экранирование, даёт `"` (в JSON `\"`). Первая одиночная `"` закрывает значение **безусловно** (без проверки следующего символа).
- Незакрытая кавычка любого типа → значение до конца события, JSON-строка всё равно корректно закрывается.

### 4.2. Значения без кавычек

Значение без кавычек тянется до следующей `,` или конца события (может содержать `=`).
Оно становится JSON-**числом**, если это **строгое JSON-число** (грамматика RFC 8259: `-?(0|[1-9]\d*)(\.\d+)?([eE][+-]?\d+)?`, длина ≤ 32) и имя поля не в списке always-string: `SearchString`, `Guid`, `UUID`. Иначе — строка.
**Решение v1.0:** `8.3.22.1704`, `1-2`, `.5`, `0.`, `1.2.3`, `007` — строки. Текущий exe выдаёт их как голые токены → невалидный JSON. Это баг (KI-2), исправление обязательно.

### 4.3. Пустые значения

`Имя=` (в конце события или перед `,`) → `""`.

### 4.4. Экранирование JSON

`"`, `\`, `\b \f \n \r \t`, прочие управляющие < 0x20 → `\u00xx` (**hex в нижнем регистре** — как у текущего exe). Байты ≥ 0x20 — как есть, UTF-8 не валидируется (KI-3: невалидный UTF-8 источника протекает в выход; допустимо в v1.0, приёмник должен быть готов).

### 4.5. Дубликаты ключей

Повтор свойства в одном событии (или свойство с именем `event`/`level`/…) → дублирующиеся ключи в JSON-объекте. Допустимо в v1.0 (KI-4); приёмник берёт последнее значение.

## 5. Порядок и детерминизм

- **Требование v1.1 (к агентам bake-off):** внутри одного файла порядок записей = порядок событий в файле.
- **Текущий exe это НЕ гарантирует** (KI-11): батчи по 2000 событий разбираются несколькими потоками и перемешиваются. Порядок восстанавливается только при `workers=1` (второй позиционный аргумент CLI = 1 → один читатель + один разборщик).
- Между файлами порядок не гарантируется никем.
- Golden-тесты гоняются с `workers=1` и сравниваются побайтно после нормализации EOL; мультифайловые кейсы сортируются харнессом по `(file_path, позиция в файле)`.

## 6. Поведение при ошибках и статистика

- Ошибка открытия/записи выходного потока → **фатальная** ошибка, ненулевой exit-код, без зависаний (текущий exe виснет — баг KI-5).
- Файл < 100 байт пропускается; счётчик пропусков обязан попадать в статистику.
- Кандидат события, у которого в заголовке нет второй запятой (нет уровня после имени события) — пропускается **со счётчиком** `parse_skips` (текущий exe теряет молча).
- Файл, который не удалось открыть/замапить, обязан попадать в счётчик ошибок и давать ненулевой exit-код в batch-режиме (текущий exe пишет строку в stderr, но рапортует успех — KI-12).
- **Требование v1.1:** прогресс/статистика/логи — только stderr (или `--stats-json`), stdout зарезервирован для данных в потоковом режиме. Текущий exe печатает прогресс в stdout (KI-13); в файловом режиме это не мешает, в потоковом — недопустимо.
