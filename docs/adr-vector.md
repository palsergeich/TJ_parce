# ADR: уроки Vector (vectordotdev/vector) для tj-агента

Статус: **принято** владельцем 2026-07-09. Источник анализа: архитектурная документация Vector
(Buffering Model, End-to-End Acknowledgements, Adaptive Request Concurrency).
Vector — конвейер наблюдаемости sources→transforms→sinks (Rust, Datadog) с file-tail
источником и ack-моделью, близкой к нашей, — его решения суть ответы на наши будущие грабли.

## Принято к реализации

### 1. Дисковый буфер follow-режима (фаза 3.6) — референс-дизайн Vector

- Сегментные append-only файлы **по 128 МиБ**, кадр = `len + CRC32 + NDJSON-строка события`
  (NDJSON — наш спецификованный формат: самоописуемость, отладимость, стабильность версий;
  повторный прогон через rich-маппинг детерминирован).
- **fsync ≤ 500 мс** (баланс скорость/долговечность), жёсткое соблюдение лимита размера
  (минимум ~256 МиБ), `when_full = block` (backpressure к читателю; ТЖ-файлы — внешний WAL).
- **Ack-точка смещается**: чекпоинт ТЖ двигается после durable-записи в буфер, а не после
  ClickHouse → агент переживает долгий простой БД без роста памяти; сегмент удаляется после
  подтверждения вставки всех его событий (внутренний reader-offset буфера).
- Повреждённый хвост сегмента восстанавливается по чексуммам (усечение до последнего
  валидного кадра); **I/O-ошибка буфера = немедленная остановка агента** (не терять молча).
- Правило сайзинга в доку (дословно из Vector): «оценивайте размер события по его
  JSON-представлению без сжатия».
- НЕ реализуем их экспериментальный overflow (память→диск каскадом): сами помечают
  как непригодный для прода и без гарантий порядка.

### 2. ARC — адаптивная конкурентность вставки (фаза 3.6)

AIMD из TCP: наращивать конкурентность/размер при хороших RTT, **мультипликативно сбрасывать**
при деградации (MEMORY_LIMIT_EXCEEDED, таймауты, рост p99 вставки). Заменяет фиксированный
пул из 3 коннектов batch-режима. Мотивация: инцидент bulk-перезаливки — сервер задыхался,
агент продолжал давить. Сырьё уже есть: гистограмма `tj_ingest_insert_seconds`,
классификация retryable-ошибок.

### 3. Карантин нераспознанного / dead letter (фаза 3.6)

`parse_skips` не только счётчик: сырые байты нераспознанных событий — в карантин
(файл `state_dir/quarantine/` с метаданными источника) для последующего разбора.
Для продукта, чьё сердце — парсер, это и диагностика, и материал развития формата.

### 4. Fanout-ack семантика Store (фаза 4)

При нескольких приёмниках событие подтверждено, **только когда подтвердили все**;
иначе — retry всем. Принимается как контракт Store-слоя до его реализации.

### 5. Эксплуатационные мелочи (фаза 3.6+)

`validate` (проверка конфига/каталогов/БД без запуска), `tap` (живая подсмотрка
распарсенных событий без CH), горячая перезагрузка конфига в службе, enrichment-таблицы
(CSV-подстановки: usr → отдел, ИБ → владелец), маркировка гарантий доставки
по компонентам в документации.

## Отклонено

- **VRL** (встроенный DSL трансформаций): наши трансформации доменные (СКД-контексты,
  SQL-нормализация), им нужны спеки и golden-тесты, а не тьюринг-полный язык в конфиге.
- **Произвольный DAG топологий**: наш конвейер линеен — это преимущество.
- **Overflow-буфер** — см. выше.
