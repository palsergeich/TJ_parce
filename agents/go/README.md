# tj-agent-go — участник bake-off (Go)

Нативный Go-порт нормализатора техжурнала 1С (`cpp_parse/count_contexts.cpp`),
байт-в-байт по [docs/format-spec.md](../../docs/format-spec.md) v1.0 (ревизия 2).
Способ парсинга по заявке (bakeoff-protocol §1.1): **native**, без FFI к C++-ядру.

## Сборка и запуск

```powershell
cd agents\go
go build -trimpath -ldflags "-s -w" -o tj-agent-go.exe .\cmd\tj-agent-go

# Контракт golden-раннера (совместим с count_contexts.exe):
.\tj-agent-go.exe <input_dir> [workers] [output.jsonl] [--no-output]

# Контракт bake-off (batch-режим):
.\tj-agent-go.exe --input <dir> --threads <N> --sink {null|file:<path>|clickhouse[:<dsn>]}
                  [--batch-rows 50000] [--batch-bytes 67108864] [--flush-ms 1000]
                  [--stats-json <path>]

# Контракт bake-off (сценарий B — follow/tail, см. раздел ниже):
.\tj-agent-go.exe --follow --input <dir> --sink clickhouse[:<dsn>] --state <dir> --stop-file <path>
                  [--poll-ms 500] [--idle-close-ms 2000] [--threads N] [--stats-json <path>]
```

Гейт корректности:

```powershell
tests\golden\run_golden.ps1 -Agent agents\go\tj-agent-go.exe
# Итого: PASS=18 FAIL=0 XFAIL=1 (mask_inside_quotes — KI-1, как у эталона)
```

## Свойства реализации

- **Детерминизм не зависит от workers** (жёстче KI-11): файлы разбираются
  параллельно, но записываются строго в порядке обхода; порядок событий внутри
  файла всегда сохраняется. Порядок файлов — по убыванию размера (стабильная
  сортировка), совместим с эталонным exe для побайтного golden-сравнения.
  На произвольном корпусе с файлами **одинакового размера** порядок файлов
  может отличаться от эталона (у exe нестабильный `std::sort`) — спека §5
  межфайловый порядок не гарантирует; содержимое по-файлово байт-в-байт
  (проверено дифф-тестом на реальных коллекциях: CallsDiag_86, EXP_86,
  Lic_86, TLockDiag).
- **Модель допуска — байтовая, вход и выход потоковые** (зеркало
  `core/src/pipeline.cpp`). Вход читается чанками по 8 МБ
  (`parser.ScanEvents`): в буфере воркера всегда лежит только текущее
  событие + чанк вперёд, кандидат маски у края чанка решается лишь при
  ≥ 64 КБ доступного префикса строки либо на конце файла (гвардейская зона,
  как в ядре) — файл никогда не находится в памяти целиком. Головной файл
  (чья очередь писаться) допускается вне бюджета: его NDJSON-чанки по 4 МБ
  писатель забирает из канала на лету — вывод не копится (стриминг).
  Остальные файлы допускаются строго по возрастанию индекса и только если
  их размер помещается в остаток бюджета `64 МБ × workers` (минимум
  256 МБ); файл больше остатка ждёт, пока сам станет головой, и тогда
  стримится с нулевой буферизацией. Дедлок невозможен: писатель всегда ждёт
  файл, который либо уже допущен, либо допустится как голова без бюджета.
  На стресс-каталоге `Mem\rphost_47988` (файлы 5.76 + 1.86 ГБ) пик RSS —
  **~52 МБ** при 10.9 с (диск-баунд; прежняя реализация с `os.ReadFile` и
  счётным окном `2×workers` давала 15–17 ГБ).
- KI-1 (ложный сплит маской внутри кавычек) воспроизводится сознательно —
  чинится в core вместе со всеми участниками, кейс остаётся XFAIL.
- Статистика успешного прогона — в stdout (как у эталона: golden-раннер на
  PowerShell 5.1 трактует stderr под редиректом как ошибку); stderr — только
  для ошибок. `--stats-json` пишет `{"events":N,"files":M,"skips":K,"bytes":B}`
  плюс расшифровку (parse_skips / small_file_skips / failed_files).
- Exit-коды: 0 — успех; 1 — аргументы/каталог/запись вывода; 2 — часть входных
  файлов не прочитана (KI-12).
- Выходные чанки переиспользуются через `sync.Pool`, входной буфер — свой у
  каждого воркера (растёт только под гигантские события вроде TLOCK): на
  скорости диска парсинг не рождает короткоживущего мусора для GC.

## ClickHouse-sink (`--sink clickhouse`)

Сценарий A bake-off (batch-ingest) поверх официального клиента
**clickhouse-go v2** (native TCP, bakeoff-protocol §1.2; зависимость
запиннена в go.mod/go.sum). События конвертируются в строки таблицы на
уровне разобранных полей (`parser.ParseEventFields`/`ScanProps`) — без
промежуточного NDJSON и повторного разбора; зеркальность обоих путей
разбора закреплена дифференциальным тестом
`TestFieldsMirrorAppendEvent`/`...GoldenInputs` (реконструкция NDJSON из
полей сравнивается с `AppendEvent` байт-в-байт, включая все golden-входы).

- **DSN**: `--sink clickhouse` → `clickhouse://localhost:9001/tj_bench`
  (по умолчанию); `--sink clickhouse:<dsn>` — явный DSN (принимаются
  написания `clickhouse://host:9001/db`, `clickhouse:clickhouse://…`,
  `clickhouse:host:9001/db`; параметры clickhouse-go в query — as is).
  Целевая таблица настраивается query-параметром `table`
  (`…/tj_bench?table=events_go`, допускается `db.имя`); по умолчанию —
  `events` в базе из DSN, т.е. продовый `tj_bench.events`.
- **Маппинг** (единый контракт участников): `timestamp` — дата+час из имени
  файла + ММ:СС.мммммм из события → `DateTime64(6)`, время без TZ трактуется
  как **UTC** (литерал текста = значение при серверной TZ UTC);
  деградированный timestamp (имя файла не `YYMMDDHH.log`) → нулевое время
  1970-01-01. `duration` → `UInt64` (насыщение на переполнении). `level` —
  строка: числовой уровень — его десятичный текст, строковый — как есть.
  `props` — **все** свойства события, имя → значение-ТЕКСТ: кавычечные
  значения после расклейки `''`→`'`/`""`→`"` (ровно содержимое JSON-строки
  NDJSON до экранирования), бескавычечные — сырой токен; многострочные
  значения сохраняют реальные `\r\n`-байты; дубликаты ключей схлопываются
  «последнее значение побеждает» (format-spec §4.5), порядок свойств
  события в `Map` сохраняется.
- **Батчи**: 50 000 строк ИЛИ 64 МБ ИЛИ 1000 мс — что наступит раньше
  (`--batch-rows/--batch-bytes/--flush-ms` переопределяют; для file/null
  принимаются и игнорируются). `async_insert` не включается, настройки
  вставки серверные по умолчанию.
- **Порядок вставки**: упорядоченного писателя здесь нет — батчи уходят по
  мере разбора, порядок **между файлами не гарантируется** (протоколу не
  нужен, MergeTree переупорядочивает по своему `ORDER BY`); порядок событий
  одного файла в потоке строк сохраняется (файл разбирается одним воркером,
  слабы и единственный батчер — FIFO).
- **Ошибки — flush-then-fail**: недоступный сервер ловится `Ping`'ом до
  начала разбора (fail-fast, exit 1); ошибка вставки по ходу фатальна —
  каждый батч отправляется синхронно и `inserted_rows` учитывает только
  подтверждённые сервером батчи, падавший батч не вставлен целиком,
  состояний «полубатча» и потерь подтверждённого не бывает; воркеры
  останавливаются сигнальным каналом, без зависаний, exit 1.
- `--stats-json` дополняется полем `inserted_rows` (на успешном прогоне
  равно `events`).
- Замер на `Diag_86` (272 МБ, 1 256 605 событий, `--threads 8`, локальный
  tj-clickhouse в Docker): медиана **≈366 тыс. строк/с** end-to-end
  (~79 МБ/с корпуса), лучший прогон 410 тыс. строк/с; count/sum(duration)/
  диапазон timestamp сверены с file-sink тем же каталогом байт-в-байт.

## Follow-режим (`--follow`, сценарий B)

Слежение за растущим каталогом ТЖ с доставкой в ClickHouse и чекпоинтами
«только после подтверждённой вставки». Единый контракт участников bake-off:

```powershell
.\tj-agent-go.exe --follow --input <dir> --sink clickhouse[:<dsn>] `
                  --state <dir> --stop-file <path> `
                  [--poll-ms 500] [--idle-close-ms 2000] [--threads N] [--stats-json <path>]
```

Сначала обрабатываются существующие файлы (с учётом чекпоинтов), затем —
непрерывный tail. Реализация — `internal/follow`: discovery (WalkDir раз в
`--poll-ms`) → воркеры-хвостовики (файл закреплён за воркером — порядок
событий файла сохраняется) → батчер chsink в Retry-режиме → ClickHouse →
ack → продвижение `committed_offset`.

- **Обнаружение**: новые `*.log` (рекурсивно) — обход каталога каждые
  `--poll-ms` (подхват нового файла < 2 с при 500 мс; в самотесте — 737 мс
  до queryable). Рост уже отслеживаемых файлов обходом НЕ детектится:
  метаданные каталога NTFS (их видит FindFirstFile/DirEntry) для файлов,
  которые писатель держит открытыми — rphost держит лог открытым весь час —
  обновляются лениво и показывают замороженный размер (S6-инцидент серии:
  вставки «прорывались» лишь после остановки писателя). Поэтому раз в
  `--poll-ms` воркер сверяет размер по СВОЕМУ открытому хэндлу (`f.Stat` →
  `GetFileInformationByHandle`, всегда точен), дремлющие файлы (хэндл закрыт
  после 30 с без роста) — точным `os.Stat` раз в 2 с, а ленивый размер
  DirEntry перед гейтом нового файла перепроверяется `os.Stat`'ом. Гейт
  `MIN_FILE_SIZE=100` перепроверяется по мере роста: 0-байтовый файл
  становится читаемым, дорастя до 100 байт. Исчезнувший и вернувшийся путь
  форсирует пересверку, а расхождение размеров по хэндлу и по пути ловит
  подмену файла под тем же именем.
- **Открытие с полным шарингом**: `CreateFile` с
  `FILE_SHARE_READ|WRITE|DELETE` (rphost держит лог открытым на запись;
  стандартный `os.Open` в Go не даёт `FILE_SHARE_DELETE`). Догнанный и не
  растущий 30 с файл «дремлет» — хэндл закрывается, переоткрывается по
  первому росту со сверкой идентичности.
- **Закрытие события в tail** (конца файла не существует): (1) пришла
  следующая строка-маска `^\d{2}:\d{2}\.\d{6}-\d+,`; (2) pending целиком
  оканчивается `\n` и нет новых данных `--idle-close-ms` → эмит по таймауту;
  (3) graceful-стоп → эмит `\n`-терминированной части. Незавершённая
  последняя СТРОКА (без `\n`) не эмитится никогда; решение «маска/не маска»
  принимается только по завершённой строке (для полной строки оно не зависит
  от следующих байтов — скан маски не пересекает `\n`). BOM пропускается
  только на оффсете 0; `date_prefix` из имени файла — как в batch.
- **Чекпоинты** (`<state>/checkpoints.json`, атомарно tmp+rename):
  `{"version":1,"files":[{"path":…,"identity":{"vol","hi","lo"},"committed_offset":N}]}`.
  Идентичность — volume serial + file index (`GetFileInformationByHandle`;
  POSIX-фолбэк dev+ino), путь — только диагностика. `committed_offset`
  продвигается ТОЛЬКО после подтверждения вставки сервером (ack батча;
  строки несут `{файл, поколение, конец события}`, порядок ack'ов одного
  файла монотонный — это min-contiguous граница). Сохранение — сразу после
  каждого ack'а (+ страховочный тик 1 с). На рестарте: резюме с
  `committed_offset`; размер < оффсета ИЛИ смена идентичности → с нуля.
  Усечение/пересоздание на лету: поколение файла инкрементируется, ack'и
  старого поколения игнорируются. Гарантия: **ноль потерь**; at-least-once
  на краше (окно дублей — один батч в полёте: kill между Send и записью
  чекпоинта).
- **Ошибки вставки**: повторы пересобранного батча с экспоненциальным
  бэкоффом 1→30 с (строки удерживаются до подтверждения; вход батчера не
  читается → чтение файлов backpressure'ится; чекпоинты стоят — потерь нет).
  После стопа число попыток ограничено (3 с бэкоффом ≤2 с), затем exit 1 с
  непродвинутым чекпоинтом. Недоступный сервер на старте — fail-fast (Ping).
- **Graceful-стоп**: появился `--stop-file` → чтение прекращается,
  `\n`-терминированные pending дренируются, финальный flush подтверждается,
  чекпоинт и `--stats-json` пишутся, exit 0. Прогресс раз в 2 с — в stderr
  (stdout чистый).
- `--stats-json` дополняется `inserted_rows`, `idle_closes`, `truncates`,
  `lag_bytes`.

Самотесты (живой tj-clickhouse): дозапись → queryable ≈1 с (+2 с idle-close
хвостового события); незавершённая строка удержана 6 с и не вставлена, после
достройки — ровно 1 запись; kill -9 посреди 20 тыс. событий + рестарт →
потерь 0, дублей 0 (резюме с чекпоинта); новый файл queryable за 0.7 с;
усечение до 0 → сброс и продолжение; остановка ClickHouse на 10+ с →
бэкофф 1/2/4/8 с, после восстановления 30/30 без потерь; initial pass
совпал с batch-режимом (501/501), повторный запуск с тем же `--state` —
0 повторных вставок. S6-сценарий (`tests/tail/generator.ps1`, писатель
держит файл открытым, 20 тыс. соб/с × 15 с): лаг по `props['GenMs']` на
каждой 2-секундной выборке 0.6–3.0 с (< 5 с), рост непрерывный, итог
300000/300000 без потерь и дублей. Golden-гейт batch-режима не тронут:
PASS=18 FAIL=0 XFAIL=1.

Ограничения: пересоздание файла с тем же путём и РОВНО тем же размером в
«дремлющем» состоянии не детектится до следующего изменения размера (на
рестарте ловится идентичностью); события-продолжения, дописанные после
idle-close своего события, отбрасываются до следующей маски (свойство
контракта, а не реализации); чекпоинт-файл пишется без fsync — kill -9
процесса безопасен, крах ОС может откатить оффсеты назад (даст дубли, не
потери).
