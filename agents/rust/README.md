# tj-agent-rs — Rust-участник bake-off

Нормализатор технологического журнала 1С → NDJSON по [docs/format-spec.md](../../docs/format-spec.md) v1.0 (ревизия 3) с вставкой в ClickHouse (`--sink clickhouse`). Только стандартная библиотека Rust — внешних крейтов нет, включая ClickHouse-синк (свой HTTP/1.1 + RowBinary, см. ниже).

## Сборка

```powershell
# cargo не в PATH — берём из ~/.cargo/bin (Rust 1.96 stable-msvc)
& "$env:USERPROFILE\.cargo\bin\cargo.exe" build --release
```

Исполняемый файл: `target/release/tj-agent-rs.exe`. Релизный профиль по bakeoff-protocol §2.1: `lto = "fat"`, `codegen-units = 1`, `panic = "abort"`.

Тесты и линт:

```powershell
& "$env:USERPROFILE\.cargo\bin\cargo.exe" test
& "$env:USERPROFILE\.cargo\bin\cargo.exe" clippy --release --all-targets   # 0 предупреждений
```

## Запуск

Три синтаксиса (все обязательны по контракту bake-off):

```powershell
# 1. Позиционный (контракт golden-раннера, совместим с count_contexts.exe)
.\tj-agent-rs.exe <input_dir> [workers] [output.jsonl] [--no-output]

# 2. Флаговый (bakeoff-protocol §1.1, batch-режим)
.\tj-agent-rs.exe --input <dir> --threads <N> --sink {null|file:<путь>|clickhouse[:<dsn>]}
                  [--batch-rows 50000] [--batch-bytes 67108864] [--flush-ms 1000]
                  [--stats-json <путь>]

# 3. Хвостовой режим (единый follow-контракт bake-off, см. раздел --follow)
.\tj-agent-rs.exe --follow --input <dir> --sink clickhouse[:<dsn>] --state <dir>
                  --stop-file <путь> [--poll-ms 500] [--idle-close-ms 2000]
                  [--threads N] [--stats-json <путь>]
```

Exit-коды: `0` — успех (в follow-режиме — остановка по `--stop-file`); `1` — ошибка аргументов/каталога/записи вывода/вставки в ClickHouse; `2` — часть входных файлов не удалось прочитать (batch-режимы).

## Синк ClickHouse

```powershell
# По умолчанию: http://localhost:8123/tj_bench/events (HTTP-порт, пользователь default)
.\tj-agent-rs.exe --input <dir> --threads 8 --sink clickhouse
# Явный DSN: http://host[:порт][/база[/таблица]]
.\tj-agent-rs.exe --input <dir> --threads 8 --sink clickhouse:http://localhost:8123/tj_bench/events_rust
```

- **Клиент** (фиксация выбора по bakeoff-protocol §1.2): официальный Rust-клиент (крейт `clickhouse` от ClickHouse Inc.) работает поверх **HTTP + RowBinary** — здесь тот же протокол, но своим минимальным HTTP/1.1-клиентом на `std::net::TcpStream` (`src/chsink.rs`). Причины: формату нужен пронос сырых байтов источника (KI-3: UTF-8 не валидируется), serde-модель крейта построена на UTF-8 `String`; ноль зависимостей и никакого async-рантайма поверх синхронного конвейера; политика батчей реализуется явно. Протокольные требования (официальные настройки вставки, `async_insert=0`) соблюдены.
- **Подключение к схеме** — на уровне разобранного события: тот же автомат `parser::parse_event`, что у NDJSON, но эмиттер `RowEmitter` кодирует RowBinary-строку напрямую (повторного разбора JSON-текста нет). Маппинг: `timestamp` → DateTime64(6) (дата+час из имени файла + MM:SS.ssssss события; стенка источника трактуется как UTC; деградированный timestamp при не-датовом имени файла → `1970-01-01 00:00:00.000000`); `duration` → UInt64 (сатурация при переполнении); `level` → String сырым токеном (числовой уровень — десятичным текстом); `props` → **все** свойства события: значение в кавычках — распакованный контент (`''`→`'`, `""`→`"`), без кавычек — сырой токен, многострочные значения сохраняют реальные байты `\r\n`, дубликаты ключей сохраняются (KI-4). В RowBinary `LowCardinality` прозрачен, `Map` — varint-счётчик + пары.
- **Батчи** — 50 000 строк ИЛИ 64 МБ ИЛИ 1000 мс (что раньше; настраивается `--batch-rows/--batch-bytes/--flush-ms`). Нарезка точная, по границам строк: полные батчи содержат ровно `batch-rows` строк; единственное превышение байт-лимита — одиночная строка крупнее лимита (уходит своим батчем). Остаток сбрасывается по концу входа.
- **Порядок строк** — батчи уходят по мере готовности («flow as parsed»), глобального межфайлового порядка нет (СУБД он не нужен — сортировку задаёт ORDER BY таблицы); события **одного файла** попадают в поток вставки в исходном порядке файла (файл разбирает один воркер, куски идут последовательно, батчер границ не переставляет).
- **Семантика сброса/ошибок** — вставка синхронная (`async_insert=0`): успех = HTTP 200 = батч закоммичен (≤ `max_insert_block_size` строк ложатся одним блоком — атомарно). Единственный повтор — пере-отправка целого батча на свежем соединении, если ПЕРЕиспользованное keep-alive-соединение оказалось молча закрытым сервером (обрыв записи запроса или EOF до первого байта ответа — сервер запрос не начинал). Любая другая ошибка фатальна: воркеры останавливаются, exit 1, в БД остаются только подтверждённые батчи. Зависаний нет: connect ≤ 5 с на адрес, чтение/запись ≤ 120 с; доступность сервера и существование таблицы проверяются ДО разбора (неверный порт/таблица → немедленный exit 1).
- **Обратное давление** — очередь воркеры→синк ограничена `2 × batch-bytes` (мин. 16 МБ); разбор не обгоняет вставку безгранично. Накладные расходы синка по памяти на `Diag_86`: peak RSS ≈ 432 МБ против ≈ 371 МБ у `--sink null` (базу съедают чанковые буферы чтения 8×32 МБ).
- **`--stats-json`** дополняется полем `inserted_rows` (подтверждённые сервером строки; при успехе равно `events`).

Замер на живом ClickHouse 24.8 (Docker, та же машина), `Diag_86` (271.75 МБ, 1 256 605 событий, `--threads 8`): медиана 3 прогонов **2.62 с** end-to-end ≈ **479 тыс. строк/с** ≈ 104 МБ/с входа; 26 батчей (25×50 000 + 6 605), средний INSERT на стороне сервера 90.6 мс. Проверка полноты: `count()`/`sum(duration)`/`min/max(timestamp)` совпали с оракулом из `--sink file` того же каталога; multiline-значения в `props` сохраняют реальные `0D 0A`.

Golden-гейт:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File ..\..\tests\golden\run_golden.ps1 -Agent .\target\release\tj-agent-rs.exe
# Итог: PASS=18 FAIL=0 XFAIL=1 (mask_inside_quotes — KI-1, ожидаемо)
```

## `--follow` — хвостовой режим (src/follow.rs)

Непрерывный ingest растущего каталога: стартовый проход по существующим файлам
(с учётом чекпоинтов), затем слежение. Только с `--sink clickhouse[:<dsn>]`;
`--state <dir>` (чекпоинты) и `--stop-file <путь>` обязательны. stdout чист,
прогресс раз в 5 с и итог — в stderr; `--stats-json` пишется на выходе
(поле `files` — все обнаруженные файлы за сессию).

- **Обнаружение** — рекурсивный опрос `--input` каждые `--poll-ms` (500)
  ТОЛЬКО ради новых `*.log`; подхват нового файла < 2 с (обнаружение будит
  воркера немедленно). Рост известных файлов метаданными не детектируется
  вовсе: воркер каждый цикл заново открывает файл и дочитывает read-ом до
  EOF — директорные метаданные NTFS для файла, удерживаемого писателем,
  замерзают до его закрытия (ловушка S6 tail-серии), а read по свежему
  хендлу видит всё. Цена — один пустой seek+read на файл за цикл. Порог
  `MIN_FILE_SIZE`=100 переоценивается по мере роста; размер для порога и
  детекции усечения берётся по открытому хендлу (он точен, в отличие от
  path-stat).
- **Шаринг** — чтение через `std::fs::File::open`: на Windows это
  `FILE_SHARE_READ|WRITE|DELETE`, файл, открытый писателем на дозапись,
  читается, писатель не блокируется (проверено против живого писателя,
  державшего файл весь тест).
- **Закрытие события в хвосте** (единые правила bake-off): (1) пришла
  следующая строка-маска `^\d{2}:\d{2}\.\d{6}-\d+,` — решение принимается,
  как только маска целиком в буфере (паттерн не пересекает `\n`), либо строка
  завершена; (2) хвост оканчивается `\n` и по файлу нет новых данных
  `--idle-close-ms` (2000) → событие эмитится; (3) graceful-стоп → дренаж
  `\n`-терминированной части. Незавершённая СТРОКА (без `\n`) не эмитится
  никогда и оффсетом не переступается. BOM — только на оффсете 0;
  `date_prefix` из имени файла — как в batch-режиме.
- **Чекпоинты** — в `--state`, по файлу-источнику: `<fnv1a64(путь)>.ckpt`,
  внутри одна строка `tjckpt \t v1 \t <volume_serial> \t <file_index> \t
  <committed_offset> \t <путь>` (identity — GetFileInformationByHandle).
  Запись атомарна (tmp + rename + fsync). Оффсет сдвигается ТОЛЬКО после
  HTTP-подтверждения вставки; вставки последовательны, оффсет файла растёт
  монотонно (мин-непрерывность). Рестарт: identity совпала и размер ≥ оффсета
  → продолжение; усечение (размер < оффсета) или пересоздание (identity
  сменилась) → чтение с нуля. Гарантия — at-least-once: потерь ноль, дубли
  возможны только в окне «ack получен, чекпоинт не записан» (краш-тест дал 0).
- **Ошибки синка** — сетевые сбои и HTTP 5xx/408/429 повторяются с backoff
  1→30 с, максимум 10 попыток (~2.5 мин); остальные статусы (нет
  таблицы/кривые данные) — фатал сразу. Пока вставка не проходит, очередь
  (2 × `--batch-bytes`, мин. 16 МБ) заполняется и чтение встаёт (обратное
  давление). После исчерпания попыток — exit 1 с нетронутыми чекпоинтами.
- **Конвейер** — `--threads` воркеров чтения (файл закреплён за воркером —
  порядок событий файла сохраняется) + поток обнаружения + синк в главном
  потоке. Батчи собираются из целых кусков (кусок ≈ 2 МБ), поэтому лимиты
  rows/bytes могут быть превышены не более чем на кусок; таймер `--flush-ms`
  взводится первой строкой батча.
- **Остановка** — появление файла `--stop-file`: дочитать до EOF, дренаж по
  правилу 3, финальная вставка, финальные чекпоинты, `--stats-json`, exit 0.

Self-тест на живом ClickHouse 24.8 (личная таблица `tj_bench.events_rust`,
та же DDL, удалена по завершении): (a) дозапись при писателе, державшем файл
открытым — 50+20 событий, 0 дублей, timestamp из имени файла; (b) строка без
`\n` удержана 6 с, после достройки — ровно одна запись с неискажённым
значением; (c) `Stop-Process -Force` на потоке 5000 событий (убит после 474
вставленных), рестарт с тем же `--state` — 5000/5000 уникальных маркеров,
**0 потерь, 0 дублей** (вставлено ровно 4526 недостающих); (d) новый файл в
новом подкаталоге виден в CH через **1424 мс** (< 2 с, дефолтные
poll/flush); (e) truncate до 0 — сброс на оффсет 0, свежие события идут,
дублей нет; (f) golden-гейт 18 PASS, `cargo test` 61 ок, clippy чистый.

S6-замер (официальный `tests/tail/generator.ps1`, писатель держит файл
открытым весь прогон, 20 000 соб/с × 15 с = 300 000 событий): лаг
`now − max(GenMs)` при опросе каждые 2 с — все замеры 240…1626 мс,
**p95 = 1626 мс** (гейт S6 — 5000 мс); темп генератора 19 997/с;
после дренажа 300 000/300 000 уникальных маркеров, **0 потерь, 0 дублей**;
exit 0.

Ограничения (сознательные): обнаружение опросом (без ReadDirectoryChangesW) —
на гигантских деревьях каталогов стоимость обхода растёт; переписывание файла
БЕЗ уменьшения размера не детектируется (нет ни усечения, ни смены identity —
стандартное ограничение tail-подхода); чекпоинты не подчищаются за исчезнувшими
файлами; `--poll-ms/--idle-close-ms` вне `--follow` игнорируются.

## Что реализовано

- **Формат** — байт-в-байт по спеке rev 3: сырые байты источника для чисел и duration (duration — только срез ведущих нулей), строгая грамматика числа RFC 8259 (длина ≤ 32, always-string: `SearchString`/`Guid`/`UUID`, к `level` не применяется), несимметричное закрытие кавычек §4.1 (`''`-экранирование с эвристикой KI-10, `"` закрывает безусловно), `\u00xx` в нижнем регистре, `file_path` из двух предков фактического пути с разделителями Windows, пропуск входного BOM, вывод без BOM, LF после каждой записи.
- **Детерминизм** — вывод байт-идентичен при любом `--threads`: порядок событий внутри файла всегда сохраняется, файлы пишутся по убыванию размера (стабильная сортировка, при равных размерах — порядок обхода: лексикографический, как `filepath.WalkDir`).
- **Память** — модель допуска по образцу `core/src/pipeline.cpp` (байтовая, а не по числу файлов). Три механизма:
  - *чанковый ввод*: файл читается кусками по 32 МБ в переиспользуемый буфер воркера, события ищутся внутри буфера, хвост от начала текущего события переносится в следующий кусок; кандидат маски у края буфера решается только при ≥ 64 КБ контекста (гвардейская зона, как в core). Файл целиком в RAM не живёт никогда;
  - *стриминг головного файла*: файл, чья очередь писаться (индекс `files_written`), допускается безусловно, его NDJSON уходит писателю кусками ≤ 4 МБ по мере готовности — вывод головного файла не буферизуется;
  - *байтовый бюджет для неголовных*: сумма размеров допущенных, но ещё не записанных неголовных файлов ≤ `max(64 МБ × workers, 256 МБ)`. Файл крупнее остатка бюджета ждёт, пока не станет головным (и тогда стримится с нулевой буферизацией). Дедлок невозможен: допуск головы от бюджета не зависит, а когда writer ждёт непринятую голову, все предыдущие файлы уже записаны и есть свободный воркер.

  Итог на стресс-каталоге `Mem\rphost_47988` (5.76 + 1.86 ГБ, `--threads 8 --sink null`): peak RSS ≈ 71 МБ (было 15–17 ГБ при чтении файлов целиком), wall ≈ 11.1–11.5 с, события и байты вывода без изменений.
- **Ошибки** — ошибка записи вывода фатальна (exit 1, без зависаний — KI-5); нечитаемый файл считается и даёт exit 2 (KI-12); файлы < 100 байт пропускаются со счётчиком; событие без второй запятой в заголовке — `parse_skips`; нет подходящих `.log` — выходной файл не создаётся, exit 0.
- **Статистика** — сводка в stdout (stderr только для ошибок — особенность PowerShell 5.1 в golden-раннере); `--stats-json` пишет `{"bytes":…,"events":…,"failed_files":…,"files":…,"parse_skips":…,"skips":…,"small_file_skips":…}` байт-в-байт с Go-агентом.

## Структура

| Файл | Содержимое |
|------|-----------|
| `src/parser.rs` | Ядро формата: маска события, заголовок, автомат свойств `parse_event` с трейтом `EventEmitter` (два эмиттера: `JsonEmitter` → NDJSON, `chsink::RowEmitter` → RowBinary), типизация, экранирование + юнит-тесты (`split_events` остаётся тест-оракулом сканера) |
| `src/scanner.rs` | Чанковое чтение с нарезкой на события: перенос хвоста, гвардейская зона 64 КБ, SWAR-поиск `\n` + тесты против оракула |
| `src/walker.rs` | Рекурсивный обход, фильтр `*.log` ≥ 100 байт, сортировка по размеру |
| `src/chsink.rs` | ClickHouse-синк: DSN, календарь/тики DateTime64, RowBinary-эмиттер, очередь с обратным давлением, батчер 50000/64МБ/1000мс, минимальный HTTP/1.1-клиент (keep-alive, chunked, ретрай стухшего соединения, классификация retryable-ошибок для follow) + тесты, включая мини-HTTP-сервер |
| `src/follow.rs` | `--follow`: хвостовой ассемблер событий (правила закрытия 1–3), identity файла (volume serial + file index), чекпоинты tmp+rename, обнаружение опросом, воркеры чтения, синк с ретраями/backoff и коммитом оффсетов по ack + юнит-тесты |
| `src/main.rs` | CLI (все три синтаксиса), пул потоков с байтовым допуском и стримингом головного файла (file-синк), конвейер воркеры→батчер→INSERT (clickhouse-синк), статистика |

Известные сознательные воспроизведения багов эталона: KI-1 (сплит маской внутри кавычек — XFAIL), KI-3 (UTF-8 не валидируется), KI-4 (дубликаты ключей), KI-10 (эвристика одинарной кавычки).
